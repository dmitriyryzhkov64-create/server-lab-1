# Онлайн-тестирование (RBPO Lab)

## Описание проекта
Сервис для проведения онлайн-тестирования, разработанный в рамках лабораторных работ по дисциплине "Разработка безопасного программного обеспечения". Система поддерживает полный цикл управления тестами: от создания контента до анализа истории прохождений.

## Основные возможности (Бизнес-операции)
1. **Создание теста с контентом**: Создание Quiz, вопросов и вариантов ответов в рамках одной транзакции.
2. **Безопасное удаление**: Удаление теста запрещено, если по нему уже существуют попытки прохождения.
3. **Редактирование теста**: Возможность обновления названия теста.
4. **Прохождение теста**: Инициализация попытки, отправка ответов и финализация с расчетом балла.
5. **Аналитика**: Просмотр истории попыток конкретного пользователя.

## Порядок выполнения API-запросов (Workflow)
Для корректной демонстрации работы проекта рекомендуется выполнять запросы в следующем порядке:

1. **Подготовка**:
    - `POST Create_user`: Создание пользователя (автора/студента).
    - `POST Create_Quiz`: Создание теста, привязанного к пользователю.
    - `GET All_Quizzes`: Получение списка тестов для уточнения ID.

2. **Администрирование**:
    - `PUT Change_Quiz`: Редактирование названия теста (доступно, пока нет попыток).
    - `DEL Delete_Quiz`: Удаление теста (доступно, пока нет попыток).

3. **Прохождение теста**:
    - `POST Start_Quiz`: Инициализация новой попытки (получение `attemptId`).
    - `POST Answer`: Отправка ответов на вопросы.
    - `POST Finish_attempt`: Завершение попытки и расчет результата.

4. **Аналитика и очистка**:
    - `GET Statistic_idUser`: Просмотр истории попыток.
    - `DEL Clear_history`: Очистка истории всех попыток (административная операция).

## Безопасность и архитектура
- **Глобальная обработка ошибок**: Реализован `GlobalExceptionHandler` для защиты от утечки системных данных.
- **Целостность данных**: Использование аннотаций `@Transactional` и проверок (`existsByAttempts`) для предотвращения нарушения ссылочной целостности базы данных.

## Инструкция по запуску
1. Установите **PostgreSQL**.
2. Настройте переменные окружения: `DB_URL`, `DB_USER`, `DB_PASSWORD`.
3. Запуск.

## Безопасность и аутентификация

Проект реализован с использованием **Spring Security** и соответствует требованиям обеспечения безопасности веб-приложений:

* **Аутентификация**: Используется протокол **Basic Auth**. Для каждого запроса требуется передача учетных данных (логин/пароль) в заголовке запроса.
* **Хранение паролей**: Пароли пользователей не хранятся в открытом виде. При регистрации выполняется необратимое хеширование с помощью алгоритма **BCrypt** (`PasswordEncoder`).
* **Контроль доступа (RBAC)**: Реализовано разделение прав доступа на основе ролей:
* `ROLE_ADMIN`: Полный доступ к управлению тестами, вопросами и пользователями.
* `ROLE_USER`: Доступ к просмотру тестов, инициализации и прохождению тестирования.


* **Валидация при регистрации**: Реализована проверка надежности пароля (минимальная длина 8 символов, обязательное наличие спецсимволов).
* **Принцип наименьших привилегий**: Доступ к критически важным операциям (удаление/редактирование контента) строго ограничен ролью `ROLE_ADMIN`.

### Тестирование безопасности в Postman

Для выполнения защищенных API-запросов необходимо:

1. Перейти во вкладку **Authorization**.
2. Выбрать тип **Basic Auth**.
3. Ввести учетные данные зарегистрированного пользователя (логин и пароль).
